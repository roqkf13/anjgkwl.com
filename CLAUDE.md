# LLM 코딩 행동 지침

LLM이 흔히 하는 코딩 실수를 줄이기 위한 행동 지침이다. 프로젝트별 지침이 있으면 본 문서와 병합해 사용한다.

이 문서의 **1·2·4절**은 안드레아 카파시(Andrej Karpathy)가 말하는 *명시 가능성(specifiability)*·*검증 가능성(verifiability)*·극단적 단순화와 방향이 같다. **3절**(정밀한 수정)은 저장소/에이전트 편집 품질을 위한 실무 규칙으로, 카파시 본인의 원문 주제는 아니다.

> Software 1.0 easily automates what you can specify.  
> Software 2.0 easily automates what you can verify.
>
> — Andrej Karpathy, [Verifiability](https://karpathy.bearblog.dev/verifiability/) (2025-11-17)

## 트레이드오프

본 지침은 속도보다 신중함을 우선한다. 사소한 작업은 상황에 맞게 완화해도 된다.

---

## 소통 방식

채팅 응답·설명·요약·커밋 메시지는 최대한 한글로 작성한다. 코드 자체(변수명·함수명 등 식별자)는 관례상 영어를 유지하되, 그걸 설명할 때는 한글로 풀어준다. 영어 기술 용어를 써야 할 때도 옆에 한글 뜻을 같이 붙인다 (예: "Skip(건너뛰기)", "Bot Fight Mode(봇 방어 기능)").

---

## 지침 저장 원칙

사용자가 "기억해줘", "항상 이렇게 해줘" 같은 지속적 지침을 줄 때는, 특정 PC의 로컬 메모리에만 남기지 않는다. 이 저장소의 `CLAUDE.md`(또는 관련 프로젝트별 `CLAUDE.md`)에 반영해서 git으로 커밋·push한다. 그래야 어느 PC에서 이 저장소를 열어도(`git pull`) 동일하게 적용된다. 저장소와 무관한 개인적 성향(예: 이 프로젝트와 상관없는 다른 작업 습관)만 로컬 메모리에 남긴다.

---

## 작업 연속성 (`_notes/`)

여러 세션·여러 PC에 걸쳐 이어지는 작업을 위해 저장소 루트 `_notes/`에 세 파일을 둔다. 모두 git으로 추적되어 어느 PC에서든 동일하게 보인다.

| 파일 | 역할 | 갱신 시점 |
|------|------|-----------|
| `_notes/HANDOFF.md` | 지금 최신 상태 스냅샷 (목표/완료된 일/진행 중인 일/다음 행동/주의점) | 매번 **덮어쓰기** — 세션을 마무리할 때("오늘은 여기까지" 등) |
| `_notes/MEMORY.md` | 결정 로그 — 방향·전략·중요한 선택과 그 이유 | 계속 **누적** — 되돌리기 어렵거나 다시 논의될 수 있는 결정이 생길 때 |
| `_notes/ERRORS.md` | 실패 노트 — 안 통했던 접근과 결국 통한 해결법 | 계속 **누적** — 두세 번 이상 헤맨 문제를 해결했을 때 |

작업을 이어갈 때는 `_notes/HANDOFF.md`를 먼저 확인한다. `CLAUDE.md`가 "지금 어떻게 해야 하는지"(현재형 규칙)를 담는다면, `_notes/`는 "왜 그렇게 했는지·무엇이 안 됐는지"(과거형 히스토리)를 담는다 — 둘을 섞지 않는다.

## 1. 구현 전 사고 (Think before coding)

가정하지 않는다. 모호함을 숨기지 않는다. 트레이드오프를 명확히 밝힌다.

구현을 시작하기 전에 다음을 지킨다.

- 자신의 가정을 **명시**한다. 불확실하면 **질문**한다.
- 해석이 여러 갈래면 임의로 고르지 말고 **대안을 나열**한다.
- 더 단순한 방법이 있으면 **제안**한다. 정당한 이유가 있으면 사용자 요청에 **반대 의견**을 말해도 된다.
- 불분명하면 **중단**하고, 무엇이 불분명한지 구체적으로 질문한다.

*(Software 1.0에서 잘 자동화되는 것은 "말로 명확히 적을 수 있는 것"에 가깝다. 요구가 흐리면 먼저 명세를 다듬는다.)*

---

## 2. 단순성 우선 (Simplicity first)

- 문제를 푸는 데 필요한 **최소한의 코드**만 쓴다. 추측으로 기능을 채우지 않는다.
- **요청되지 않은 기능**을 넣지 않는다.
- 일회성 작업에 **불필요한 추상화 층**을 만들지 않는다.
- **요청되지 않은** 유연성·설정 가능성을 미리 넣지 않는다.
- **현실적으로 일어나지 않는** 시나리오만을 위한 방어 코드에 시간을 쓰지 않는다.
- 200줄을 50줄로 줄일 수 있으면 **다시 쓴다**.
- "시니어 엔지니어가 보기에 이 코드가 지나치게 복잡한가?"라고 자문한다. **그렇다면** 단순화한다.

*(카파시가 예로 드는 것처럼, "더 이상 줄일 수 없을 때까지" 본질만 남기는 방향을 목표로 한다.)*

---

## 3. 정밀한 수정 (Surgical changes)

필요한 부분만 고친다. **자신이 만든 코드**의 뒷정리는 한다.

기존 코드를 편집할 때:

- 인접한 코드·주석·포맷을 **임의로** 고치지 않는다.
- **망가지지 않은** 부분을 리팩터링하지 않는다.
- 스타일이 마음에 안 들어도 **기존 스타일**을 따른다.
- 작업과 무관한 데드 코드를 보면 **보고만** 하고, 요청 없이 **지우지 않는다**.

자신의 수정 때문에 불필요해진 것만 정리한다.

- 불필요해진 import·변수·함수는 **제거**한다.
- 원래부터 있던 데드 코드는 **요청이 없으면 그대로** 둔다.

**변경 품질 기준:** diff의 모든 줄이 사용자 요청과 **직접적으로** 연결되어야 한다.

---

## 4. 목표 중심 실행과 검증 가능성 (Goal-driven, verifiable work)

성공 기준을 **정의**하고, **검증**될 때까지 좁혀 간다. 작업을 가능한 한 **검증 가능한** 목표로 바꾼다.

- "유효성 검사 추가" → 잘못된 입력에 대한 **테스트 작성 → 통과**
- "버그 수정" → **재현** → 테스트(또는 재현 스크립트) → **통과**
- "X 리팩터링" → 리팩터링 전후 **테스트 통과** (또는 동등 동작을 증명하는 최소 확인)

다단계일 때는 짧게라도 계획을 쓴다.

1. [단계] → 검증: [무엇으로 확인하는가]
2. [단계] → 검증: [무엇으로 확인하는가]
3. [단계] → 검증: [무엇으로 확인하는가]

성공 기준이 모호하면(예: "그냥 되게") 같은 질문이 반복된다. **재시도·자동 피드백**(테스트, 린트, 재현 절차)이 있을수록 작업은 Software 2.0 관점에서 *검증 가능*에 가깝고, 에이전트가 안정적으로 기여하기 쉽다.

## 이 지침이 잘 쓰이고 있는지

- diff에 **불필요한 변경**이 줄었는가
- 복잡도 때문에 **같은 일을 다시 짜는** 일이 줄었는가
- 구현 전 **질문과 의사결정**이 분명해졌는가

## Git 브랜치 전략

이 저장소는 `main`, `henry`, `abi` 세 브랜치를 함께 쓴다. 브랜치를 동기화(merge)할 때는 서로 갈라졌는지(diverge) 여부로 방법을 나눈다.

- **한쪽만 앞서있고 나머지는 그 조상인 경우** (예: abi에만 새 커밋, henry/main은 그대로): `git merge --ff-only` + push로 바로 맞춰도 된다. 충돌 가능성이 없는 단순 fast-forward이기 때문이다.
- **두 브랜치 이상이 각자 독립적으로 커밋을 쌓아 갈라진 경우** (예: abi와 henry가 각자 다른 작업을 진행 중): 브랜치끼리 직접 merge하지 않는다. 각각 `main`으로 PR을 올려 순서대로 merge하고(충돌 여부가 PR에서 명확히 드러남), 그 다음 각 브랜치가 `main`에서 다시 당겨와 서로 맞춘다. `main`은 PR로 검증된 상태를 유지하는 브랜치이므로, 이 경우엔 브랜치 간 직접 merge보다 PR 경유가 안전하다.

병합 전 `git merge-base --is-ancestor <branch> <other>`로 fast-forward 가능 여부를 먼저 확인한다.
