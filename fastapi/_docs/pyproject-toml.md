# A2A 멀티 에이전트 프로젝트 — 설계 검토에서 건진 장점 정리

A2A(에이전트 간 통신) 포트폴리오 프로젝트를 구상하면서 여러 초안(`pyproject.toml` 하네스 스펙 포함)을 검토했다. 아직 실행 계획은 아니고, 실제로 채택할 만하다고 판단된 설계 원칙만 모아둔 메모다. 나중에 이 프로젝트를 시작할 때 체크리스트로 쓴다.

## 우리 실제 환경 (참고 — 초안들에 잘못 적혀있던 값 정정)

- GPU: RTX 3050, **VRAM 8GB** (`nvidia-smi`로 직접 확인함. 초안에 6GB로 반복해서 적혀있었는데 오류였음)
- 온프레미스에서 이미 Ollama로 **EXAONE 3.5 7.8B**를 구동 중 (`fastapi/.env`의 `OLLAMA_MODEL=exaone3.5:7.8b`) — RAG 파이프라인(`apps/rag`)이 실제로 이걸 쓰고 있음
- 인프라 이미 보유: Cloudflare Tunnel(`anjgkwlcom-cloudflared-1` 컨테이너), Neo4j(`anjgkwlcom-neo4j-1` 컨테이너) — 새로 구축할 필요 없음

## 채택할 설계 원칙

### 1. 기존 Ollama 서빙을 재사용한다

GPU 추론을 `transformers` + `bitsandbytes`로 새로 구현하지 않는다. 이미 안정적으로 동작 중인 Ollama HTTP API를 그대로 호출한다. 이미 검증된 걸 버리고 새로 만드는 건 불필요한 재작업이다.

### 2. 패키지는 uv workspace로 묶지 않고, 배포 단위별로 독립시킨다

온프레미스 에이전트와 AWS 에이전트는 물리적으로 다른 서버에 배포되므로, 하나의 workspace로 묶기보다 각 디렉터리가 독립적인 `uv sync` 단위가 되는 게 맞다. 배포 대상이 다르면 패키징 단위도 다르게 간다.

### 3. AI/GPU 의존성이 필요 없는 에이전트(AWS 라우터)는 정말로 아무것도 안 넣는다

`ollama`, `neo4j`, `torch` 같은 의존성을 AWS 쪽 `pyproject.toml`에 절대 넣지 않는다. "그냥 안 넣으면 됨"으로 끝내지 않고, **실제로 설치 안 됐는지 검증하는 명령어**를 완료 기준에 포함한다.

```bash
uv pip list | grep -E "ollama|neo4j" && echo "FAIL: 금지 의존성 발견" || echo "ok"
```

### 4. 에이전트 간 메시지 스키마는 공통 패키지 하나로 분리한다

`a2a_shared` 같은 별도 패키지에 pydantic 스키마를 두고, 각 에이전트가 이걸 의존성으로 가져다 쓴다. 스키마를 에이전트마다 복제하지 않는다 — 단일 소스.

### 5. 데이터 접근 경로를 하나로 고정한다

 DB(Neo4j)는 온프레미스 에이전트만 직접 연결하고, AWS 쪽은 반드시 MCP 경유로만 접근한다. "누가 DB에 직접 붙는지"가 코드만 봐도 명확해야 한다.

### 6. 표준 스펙을 그대로 안 따르면 문서에 명시한다

MCP 위에 A2A 개념을 자체 구현하는 거라 진짜 A2A 스펙(AgentCard, task lifecycle 등)과는 다르다. README에 이걸 숨기지 않고 명시한다. 나중에 이 프로젝트를 보는 사람(혹은 미래의 나)이 표준 스펙인 줄 오해하지 않게.

### 7. 한 번에 다 만들지 않고, 검증 가능한 단위로 쪼갠다

전체 실행 계획을 한 번에 밀어붙이지 않고, "패키징 골격만 먼저 만들고 `uv sync`/import 확인" 같은 작은 단위로 쪼개서 각 단계가 실제로 되는지 확인하고 다음으로 넘어간다.
